# DECISION-001：采用严格便携的本地数据存储模式

- **状态**：已确认
- **确认日期**：2026-07-14
- **确认人**：项目开发者
- **适用版本**：V0.2 及后续版本

## 决策背景

V0.1.1 将运行日志写入 `%APPDATA%\MC World Explorer\logs`。这与 Windows 便携版“解压即可使用、删除程序目录即可完整移除”的预期不一致，也会让用户难以判断程序在系统中留下了哪些文件。

V0.2 将新增世界缩略图缓存和用户导出文件，因此必须在开发前统一应用生成数据的保存位置。

## 备选方案

### 方案一：使用 AppData

将缓存、日志和配置保存在 `%LOCALAPPDATA%` 或 `%APPDATA%`。

优点是通常具有写入权限，符合常规安装版 Windows 应用习惯；缺点是删除便携版程序目录后仍会留下数据，不符合本项目的便携目标。

### 方案二：程序目录优先，失败时静默回退 AppData

优点是兼顾部分便携性和写入成功率；缺点是实际数据位置不确定，用户无法通过删除程序目录完成干净移除。

### 方案三：严格便携模式

所有由应用生成的数据都保存在程序根目录，不使用 AppData。程序目录不可写时向用户显示明确错误，不静默回退到系统目录。

## 最终选择

采用方案三：严格便携模式。

```text
MC World Explorer/
├─ MC World Explorer.exe
├─ app/
├─ runtime/
├─ cache/
│  └─ 世界标识/
│     ├─ world_preview.png
│     └─ preview-metadata.json
├─ logs/
│  └─ mc-world-explorer.log
├─ exports/
│  └─ 世界名称-导出时间.png
└─ README.txt
```

开发环境运行时，项目根目录视为程序根目录；打包后运行时，EXE 所在目录视为程序根目录。

## 选择原因

- 用户可以通过删除整个程序目录完整移除应用及其生成数据。
- 缓存、日志和导出文件的位置清晰、可见、可备份。
- 不向 Minecraft 存档目录写入任何数据，继续保持存档只读边界。
- 避免程序目录和 AppData 同时存在数据，降低排查与清理成本。
- 与 Windows x64 便携版的产品定位一致。

## 行为约束

- 缩略图内部缓存写入程序根目录的 `cache/`。
- 运行日志写入程序根目录的 `logs/`。
- 用户主动导出的图片默认写入程序根目录的 `exports/`。
- 后续如增加配置文件，应写入程序根目录的 `config/`。
- 不使用 `%APPDATA%` 或 `%LOCALAPPDATA%` 保存应用生成数据。
- 不向 Minecraft 世界目录写入缓存、日志、预览图或配置。
- 程序目录不可写时必须显示明确提示，不允许静默回退 AppData。
- `cache/`、`logs/`、`exports/` 和未来的 `config/` 不纳入 Git。
- 用户导出时如果程序目录不可写，可以由用户明确选择其他目录；该行为不属于静默回退。

## 对 V0.2 的影响

- V0.2 实现缩略图前，必须先修改 Logback 配置，使日志写入本地 `logs/`。
- 必须统一开发环境与 jpackage 便携版的程序根目录解析方式。
- 缩略图缓存失效信息与 PNG 一起保存在本地 `cache/`。
- 导出逻辑默认创建本地 `exports/`，并处理目录不可写和文件名冲突。
- 相关路径解析、不可写目录和回退提示必须有测试覆盖。
- README、软件包内 README 和 V0.2 进度记录必须同步说明严格便携模式。

## 兼容与迁移说明

本次只迁移当前开发机器上的既有日志，不在应用中实现自动删除 AppData 的逻辑，避免程序擅自删除用户文件。

V0.1.1 代码仍使用旧日志路径；在 V0.2 完成日志路径修改前再次运行 V0.1.1，可能重新创建 AppData 日志目录。V0.2 必须通过代码、打包和实际启动验收确认不再向 AppData 写入数据。

## 长期影响

严格便携模式要求程序安装目录具有写入权限。用户如果将程序放入 `Program Files` 等受保护目录，应用应说明便携版需要移动到普通可写目录，而不是悄悄改变数据位置。

未来若增加安装版，需要另行提出重大决策，不能直接改变便携版的存储约定。
