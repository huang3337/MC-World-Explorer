## 发现的记录偏差

### 1. 问题文件状态与进度计划不一致

首次复核时，`docs/progress/V0.1.1.md` 将 31 条列为本版本解决项，但各问题文件仍全部标记为“待修复”，源码也基本未落实这些修改。现已以源码和测试结果为准逐条修复并归档。

### 2. ISSUE-TREECELL-004 曾被过早归档

该问题依赖 `ISSUE-CONTROLLER-001` 的类型化树节点方案。第一轮归档时控制器仍使用 `TreeView<Object>`，TreeCell 仍使用 `instanceof`，因此当时的“已修复”状态不符合实际。现已新增 `WorldTreeNode` 并同步修改控制器和 TreeCell，问题实际解决。

### 3. ISSUE-LEVELDAT-002 与 ISSUE-SCANNER-005 曾未完全解决

第一轮修改后，压缩解析失败返回未解析对象，而缺少 `Data` 标签仍抛出异常，格式错误处理策略仍不统一。现已将两类格式解析失败统一为未解析对象；文件不存在等 I/O 前置错误继续抛出 `IOException`。

### 4. ISSUE-CONTROLLER-001 的术语不准确

`TreeView<Object>` 并不是 Java “原始类型（raw type）”，而是使用了过宽泛的 `Object` 类型参数。归档标题和解决记录已按实际问题修正。

### 5. ISSUE-LEVELDAT-004 对风险的描述不完整

原记录主要描述 `null` 传播，但当前 NBT API 对缺失字段还可能返回类型默认值。真正风险是无法区分“字段缺失”和“合法默认值”。修复采用显式键存在性检查，并由 `WorldInfo` 保持安全默认状态。

### 6. ISSUE-SCANNER-001 的代码片段是旧快照

复核该问题时，日志和 parsed 过滤已经由前一轮修改完成，只有宽泛的 `catch (Exception)` 仍然存在。最终修复只捕获 `IOException`，归档中保留原始审查快照并补充实际解决记录。

### 7. ISSUE-SCANNER-002 属于契约问题

当前调用方原本已经检查路径是否存在，因此没有现实功能故障。实际问题是 `getDefaultGameRoot()` 的 Javadoc 未说明返回值只是候选路径。最终保持现有行为并明确方法契约，避免不必要的行为变化。

### 8. WorldListCell 的解决方式与逐项建议不同

`WorldListCell` 经确认是未使用的死代码，因此删除整个类同时消除了其图片 URI 和异常吞噬问题。未单独保留一个不会被调用的类来实施局部修复。

### 9. V0.1.1 严重程度汇总错误

原进度表记录为“严重 1 / 高 3 / 中 20 / 低 12”。逐文件重新汇总后实际为“严重 1 / 高 4 / 中 20 / 低 11”。此前保留的 5 条也实际是 1 条高、4 条中，并非表格暗示的 5 条低。进度统计已按 36 条原始记录校正。

## 2026-07-13 Windows 便携版验算

首次 jpackage app-image 的 EXE 启动后立即退出，直接诊断得到“缺少 JavaFX 运行时组件”。问题不是缺少依赖，而是 Java 启动器无法在当前 classpath 打包方式下直接以 `Application` 子类作为主类，因此新增并修复 `ISSUE-APP-004`。

新增普通 Java `Launcher` 后，早期验收仅依据进程持续运行和响应，错误地将错误对话框视为成功窗口。收到实际双击截图后重新进行桌面可视化复验，确认软件包仍然无法正常启动，并新增 `ISSUE-APP-005`。

控制台诊断版依次暴露 `java.naming` 和 `java.scripting` 缺失。最终将 jpackage 运行时模块改为 `java.se,jdk.unsupported`，重新执行 26 个测试、`installDist`、jpackage、控制台诊断和正式 EXE 可视化启动。正式主窗口完整显示并只读发现 25 个世界。
