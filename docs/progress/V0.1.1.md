# V0.1.1 代码质量与存档兼容性加固开发记录

## 开发目标

我在 V0.1 已有功能基础上完成代码审查问题整改、构建与启动环境补全，并修复大型整合包 `level.dat` 无法识别的问题。

本版本不开发 Region 地图、缩略图或其他 V0.2 功能，也不对用户存档执行任何写入操作。

## 当前开发阶段

V0.1.1 基线已于 2026-07-12 合并并推送到远端 `main`。2026-07-13 发布后审查发现的详情字段、目录选择和 UI 阻塞问题已在本地 `main` 工作区修复并完成发布前验收，尚待提交同步；当前不制作 V0.1.1 软件包。

## 开发范围

### 代码质量加固

- 引入 SLF4J + Logback，记录应用、解析器和扫描器异常。
- 增加全局未捕获异常处理。
- 统一 `LevelDatReader`、`WorldScanner` 和 UI 的解析失败状态。
- 提取 `GameType` 枚举，移除 UI 层的数字模式映射。
- 加强 `WorldInfo` 的空值、缺失字段和合法默认值表达。
- 使用类型化 `WorldTreeNode` 替代过宽泛的 `TreeView<Object>` 节点。
- 删除未使用的 `WorldListCell`，整理 FXML、CSS 和资源字符串。
- 缓存世界图标，收窄扫描器捕获的异常类型。

### 构建与启动环境

- 补齐 Gradle 8.7 Wrapper：`gradlew`、`gradlew.bat`、Wrapper JAR 和配置文件。
- 新增 `run-app.bat`，用于在 Windows 中双击启动应用，并在失败时保留终端错误信息。
- 统一 VS Code Java 与 Gradle 使用 Temurin 21.0.11，并重建失效的 Java 工作区缓存。

### 整合包存档兼容性

- 对 `D:\MC\.minecraft\versions` 执行只读样本检查。
- 使用 16 MiB 有界 `BinaryTagIO.Reader` 解析大型 `level.dat`。
- 三种压缩格式尝试共用相同的读取限制。
- 增加大型 NBT、超限拒绝和真实目录集成测试。

## 开发过程中遇到的问题

### 1. 问题记录与源码实际状态不一致

最初的进度记录把 31 个问题列为计划解决项，但对应 issue 文件仍标记为“待修复”，源码中多数修改也没有落实。我重新以源码、测试和运行结果为准验算全部记录，不能仅依据文档状态归档。

首次整改后仍有 5 个问题未被纳入完成范围。复核发现它们是 1 个高等级和 4 个中等级问题，而不是可暂缓的低等级问题，因此我继续完成了这 5 个问题。

### 2. 部分问题曾被过早判定为已解决

- `ISSUE-TREECELL-004` 依赖类型化树节点。仅修改 TreeCell 不能解决根因，我新增 `WorldTreeNode` 并同步修改控制器后才完成验收。
- `ISSUE-LEVELDAT-002` 与 `ISSUE-SCANNER-005` 在第一轮修改后仍存在格式失败策略不一致的问题。我统一了降级对象和扫描过滤逻辑。
- `ISSUE-LEVELDAT-004` 的原记录只强调空值传播，但 NBT API 还可能为缺失字段返回类型默认值。我增加显式键存在性检查，以区分“字段缺失”和“合法默认值”。

### 3. 部分审查记录的术语或建议与实际情况不符

- `TreeView<Object>` 不是 Java 原始类型，而是使用了过宽泛的 `Object` 类型参数。
- `ISSUE-SCANNER-001` 的代码片段是审查时快照；复核时其中一部分已经由前序修改解决，最终只需收窄异常捕获。
- `ISSUE-SCANNER-002` 属于方法契约不清，不是当前调用路径上的功能故障。我补充契约说明，没有制造不必要的行为变化。
- `WorldListCell` 是未使用的死代码，因此三个相关问题通过删除该类统一解决，而不是继续维护一个没有调用方的实现。
- 原严重程度汇总写成 1/3/20/12，逐条重算后原始 36 个问题应为 1/4/20/11。

### 4. 项目缺少可直接使用的 Gradle Wrapper

项目最初只有 Wrapper 配置残留，缺少启动脚本和 `gradle-wrapper.jar`，系统 PATH 中也没有全局 Gradle。补齐 Wrapper 时还发现 `.gitignore` 后部的 `*.jar` 规则会再次忽略 Wrapper JAR，因此增加了明确例外。

官方 Gradle 分发地址在当前网络环境下载超时，并产生了未完成的临时下载；腾讯镜像可访问且与本机已有 Gradle 8.7 缓存一致，因此最终保留镜像地址。首次切换到官方地址也导致 Wrapper 无法复用现有缓存，这不是 Java 源码错误。

### 5. VS Code 的 Gradle 和 Java 错误来自工具环境

VS Code 首先报告无法建立本机回环连接。重启后 Gradle 已能发现 37 个任务，但旧版 Red Hat Java 1.54.0 工作区缓存继续引用已经不存在的 `init.gradle`。我将项目 Java、Gradle JVM 和运行时统一到 Temurin 21.0.11，备份旧缓存并让 Java 1.55.0 重建工作区后恢复。

这些错误发生在 Gradle/Java 扩展连接和缓存层，不是项目源码编译错误。

### 6. 大部分整合包世界被扫描到但解析失败

我对 `D:\MC\.minecraft\versions` 中的 19 个实例目录进行只读检查：17 个包含 `saves` 目录，15 个包含有效世界，共发现 24 个有效 `level.dat`。全部文件均为有效 GZIP，但旧实现只能识别其中 2 个较小文件，另外 22 个因 NBT 默认读取额度不足失败。

失败日志最后显示的 `incorrect header check` 来自后续 ZLIB 尝试，容易遮蔽首次 GZIP 尝试中的读取额度错误。解压后的失败样本约为 138 KiB 至 6.3 MiB，因此我采用 16 MiB 有界 Reader，而没有使用无限 Reader。这个选择既覆盖当前样本，也保留异常输入的内存安全边界。

### 7. GitHub 发布流程未完全闭环

当时发布提交和分支推送已经成功，但当前环境没有 `gh` 命令，GitHub 连接器也无法读取该仓库并返回 404，因此没有自动创建 Pull Request。该阶段远端发布分支与本地提交一致，V0.1.1 尚未进入 `main`。

后续处理：开发者确认直接同步到 `main`，V0.1.1 已于 2026-07-12 推送到远端 `main`，临时发布分支随后删除。2026-07-13 决定先完成发布前用户流程修正与验收，再评估软件包、版本标签和 Release。

### 8. V0.1 路线图详情字段未完全显示

2026-07-13 整体审查发现，解析器虽然已经读取游戏时间和出生点坐标，但详情面板没有显示；路线图中的创建时间也没有明确数据来源。

开发者确认采用存档文件夹的文件系统创建时间。界面明确标注为 `Folder Created`，避免将其误写成 Minecraft 内部的精确世界创建时间。游戏时间按 20 ticks/秒换算为可读时长；出生点只有 X/Y/Z 三个字段全部存在时才显示，从而区分字段缺失和合法的 `(0, 0, 0)`。

### 9. 自定义目录选择与扫描契约不一致

界面原本提示选择存档目录，但控制器只按游戏根目录扫描。直接选择 `saves`、`versions`、单个实例或世界目录时会返回空结果。新增 `scanSelectedPath()` 后，扫描器根据实际目录结构选择只读扫描方式，不再要求用户理解内部路径契约。

### 10. 存档扫描阻塞 JavaFX 主线程

应用原本在初始化和目录切换时同步遍历、解压并解析全部 `level.dat`。现改用 JavaFX `Task` 和 daemon 工作线程执行扫描，扫描期间禁用目录按钮并显示进度，完成后显示世界数量、空结果或失败状态。

## 存档安全边界

- 应用和集成测试仅以只读方式打开 `level.dat`。
- 本版本不修改、复制、移动或删除真实存档。
- 本版本不解析或写入 `region/*.mca`、`playerdata/*.dat` 等后续版本内容。
- 大型 NBT 使用 16 MiB 上限，不允许无限制读取。
